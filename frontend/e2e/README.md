# Интеграционные e2e-тесты (Playwright)

Проверяют, что фронтенд и бэкенд работают вместе, и что ключевой путь
бронирования проходит от начала до конца в реальном браузере (Chromium).

## Что тестируется (пользовательские сценарии)

1. **Основной путь бронирования (гость).** `booking.spec.ts`
   Лендинг → «Записаться» → выбор типа события → выбор дня в календаре →
   выбор свободного слота → «Продолжить» → форма (имя + email) →
   «Подтвердить бронирование» → страница «Бронирование подтверждено» с
   корректными деталями (тип события, имя, email).

2. **Админ видит созданную бронь.** `admin-bookings.spec.ts`
   После брони `/admin/bookings` содержит строку с email/именем гостя и
   нужным типом события.

3. **Админ отменяет бронь.** `admin-bookings.spec.ts`
   Кнопка «Отменить» → тост «Бронирование отменено» → строка исчезает.

4. **Негатив: слот становится занятым (double-book).** `booking.spec.ts`
   После брони повторный заход на тот же тип/день показывает слот как
   «Занято» (некликабелен) — подтверждает, что бэкенд реально пересчитал слоты.

## Как запускать

Из каталога `frontend/`:

```bash
npm install
npx playwright install chromium   # разовая установка браузера
npm run test:e2e
```

Playwright сам поднимает оба сервера (`webServer` в `playwright.config.ts`):

- **бэкенд** — `e2e/scripts/start-backend.sh` создаёт `backend/.venv` (если нет),
  ставит зависимости из `requirements.txt` и запускает Flask на `:3000`;
- **фронтенд** — `npm run dev` (Vite на `:5173`) с `VITE_API_URL=http://localhost:3000/api`.

Полезное:

```bash
npm run test:e2e:ui       # интерактивный UI-режим
npm run test:e2e:report   # открыть HTML-отчёт после прогона
```

## Замечания по детерминизму

- Тесты идут **последовательно** (`workers: 1`): бэкенд хранит данные в памяти
  и общий стейт нельзя делить между параллельными воркерами.
- Каждый тест создаёт бронь с **уникальным email** и не зависит от порядка.
- Используется **завтрашний день в UTC** — гарантированно внутри окна 14 дней
  и без отфильтрованных «прошедших» слотов.
- Браузер запускается с `timezoneId: 'UTC'`, чтобы UTC-сетка слотов бэкенда и
  фронтенда совпадала с тем, что видит тест.
- Бэкенд in-memory: при перезапуске стейт сбрасывается. Между локальными
  прогонами брони сохраняются, но тесты к этому устойчивы (уникальные email,
  выбор «первого свободного» слота).
